## Context

The illustrated cake on the landing page (`palermosplash.wpenginepowered.com`) is built from an image asset plus five clickable zones (buttons) generated by `palermo-cake/palermo-cake.js`. The button DOM elements are positioned by CSS in `palermo-cake/palermo-cake.css` lines 58-101, where `button-1` sits at the lowest `bottom` value (bottom of the cake) and `button-5` sits at the highest (top of the cake). The button-to-data wiring uses `find(d => d.id === i)` in `palermo-cake.js:26`, so each numeric id picks the matching entry from `PALERMO_CAKE_DATA` in `palermo-cake-data.js`.

The product intent is Cafe on top, Custom Cakes on the bottom. The current data file assigns `id: 1` to Cafe and `id: 5` to Custom Cakes, which inverts the visual order relative to the product intent. There is no spec today that documents the layer-to-position mapping, which is what allowed the inversion to ship without being caught at review time. The existing `layer-data` spec documents data file structure but does not govern visual placement.

This change reassigns the two end-entry ids so that the product intent is satisfied, and introduces a new `cake-visual-layout` spec to lock in the layer-to-position contract going forward.

## Goals / Non-Goals

**Goals:**
- Render Cafe on the top layer and Custom Cakes on the bottom layer of the cake visual.
- Keep the data file's structural contract intact (same fields, same array length, ids in 1..5).
- Codify the layer-to-position mapping in a new spec so the inversion cannot recur.
- Verify the fix end-to-end in a browser (hover, click, popup, navigation).

**Non-Goals:**
- Editing `palermo-cake/palermo-cake.css`. CSS positions for `button-1` (bottom) and `button-5` (top) stay as-is.
- Changing per-layer content (icons, shortName, fullName, description, url) on any entry.
- Modifying the `layer-data` spec wording. Its drift after this change is tracked as a follow-up task and will be addressed in a separate OpenSpec change.
- Reordering the array entries in the data file. The array is not rendered in source order; only id bindings matter.

## Decisions

### Decision 1: Reassign ids on the two end entries instead of editing CSS

The visual order inversion could be fixed two ways: swap the CSS `bottom` values, or swap the id bindings in the data file. We pick the data-file route because:

- The CSS positions are a stable contract with the cake image asset. The five hit zones are sized and placed to match the visible layers in the image. Flipping the CSS values would desync the labels from the artwork if the image's natural top/bottom ever shifts.
- The button-to-data wiring uses `find(d => d.id === i)`, so the id is the single source of truth for which label sits on which visual slot. Swapping two ids in the data file re-routes the labels with the smallest possible surface area (two single-digit changes).
- The data file change is reviewable as a pure data diff; a CSS swap would touch the styling rules and risk subtle visual regressions.

Alternatives considered:

- **Swap CSS `bottom` values for the 5 button rules in each orientation (horizontal + vertical).** Rejected: decouples CSS positions from the image asset, requires changing 10 values, and conflates layout with content.
- **Reorder the array entries so the first entry is Cafe and the last is Custom Cakes, keeping ids the same.** Rejected: the data file is not rendered in source order, so reordering has no visual effect and only obscures the actual change.
- **Reverse the meaning of ids 1 and 5 globally (treat id 1 as top, id 5 as bottom) without changing data.** Rejected: would require editing the `layer-data` spec and the placeholder example, and would also leave the data file's id 1 = Cafe claim intact, perpetuating the contradiction rather than resolving it.

### Decision 2: Add a new `cake-visual-layout` spec instead of patching `layer-data`

The current `layer-data` spec (`openspec/specs/layer-data/spec.md`) documents the data file's structural contract (array shape, fields, enqueue order, placeholders). It does not, and should not, govern the visual position of each layer on the page — that is a layout concern, not a data concern. Mixing them risks the data spec growing layout responsibilities it was not designed for.

A separate `cake-visual-layout` spec gives the layout contract its own home with its own scenarios (top layer is Cafe, bottom is Custom Cakes, middle order is fixed, button-to-id wiring is by id not index). This keeps each spec focused and reviewable.

Alternatives considered:

- **Patch the `layer-data` spec to add a "Layer positions" section.** Rejected: blurs the boundary between data shape and page layout, and would also require reconciling the contradictory top/bottom wording already in that spec (lines 17-18, 82-115) as part of this change.
- **Patch the `layer-data` spec and the new layout spec together.** Rejected: out of scope for a bug fix; the spec wording reconciliation is a separate concern that warrants its own change so it can be reviewed and archived on its own merits.

### Decision 3: Treat CSS button positions as a stable, unchanging contract

The CSS rules `.horizontal .cake-button#button-N { bottom: ...vh }` and the vertical counterpart are a load-bearing contract: button-1 is the bottom layer, button-5 is the top layer. We document this in the new spec (Requirement: Cake button positioning remains stable) so any future reviewer or AI assistant is steered away from "fixing" the order by editing CSS.

## Risks / Trade-offs

- **[Risk] Spec drift in `layer-data` after the change** → The `layer-data` spec at `openspec/specs/layer-data/spec.md` lines 17-18 says "first entry (index 0) SHALL correspond to the top layer (Custom Cakes)" and "last entry (index 4) SHALL correspond to the bottom layer (Cafe)". After this change, the array's first entry becomes `id: 1` (Custom Cakes) which is at the BOTTOM visually. The spec's claim that Custom Cakes is the top layer is wrong w.r.t. the new product intent. → Mitigation: a follow-up task in `tasks.md` records the drift; a separate OpenSpec change will reconcile the wording.
- **[Risk] External consumer depends on `id: 1` = Cafe** → Any code outside this repo that reads `PALERMO_CAKE_DATA` and treats `id: 1` as Cafe (or `id: 5` as Custom Cakes) will break. → Mitigation: the data array is enqueued only on the landing page (per `AGENTS.md` "All asset URLs point to WP media library, not local files" and the single-feature scope), and there is no other consumer in the repo. We grep for `PALERMO_CAKE_DATA` and `id: 1` / `id: 5` references during apply to confirm no other code path is affected.
- **[Risk] Misalignment with the cake image** → If the cake image's natural top/bottom were ever flipped, the current fix would put Cafe at the bottom of the artwork. → Mitigation: the image asset is part of the WP media library and is not modified by this change. Visual verification in the browser (tasks 2 and 3) confirms the alignment before archive.
- **[Trade-off] Minimal-touch change accepts a temporary spec inconsistency** → We deliberately do not patch the `layer-data` spec in this change. The inconsistency is documented and tracked but not resolved. This keeps the bug fix small and reviewable, at the cost of leaving a small piece of spec debt until the follow-up change lands.
- **[Trade-off] Button hit zones stay where they are; only the labels move** → The hit zones are not rebuilt or resized. If a layer's label is much wider than the previous one, it could overflow the hit zone. Inspection of the data file shows the longest label ("365 Online Shop") moves from id 4 to id 2, and "Wholesale" moves from id 2 to id 4. The CSS width of 53vw is generous relative to the longest label, so this trade-off is acceptable.

## Migration Plan

- Deploy: copy the updated `palermo-cake/palermo-cake-data.js` to the WP environment via the existing child-theme / custom-plugin enqueue path. No cache busting beyond normal WordPress cache invalidation.
- Rollback: revert the two id values in `palermo-cake/palermo-cake-data.js` (id 1 → id 5 on the Custom Cakes entry, id 5 → id 1 on the Cafe entry). No data migration, no schema change, no external API impact.
- No database changes, no new dependencies, no build step.

## Open Questions

- Should the `layer-data` spec reconciliation be scheduled now or after this change ships? Tracked as follow-up; not blocking.
- Is the cake image asset guaranteed to keep Cafe on top and Custom Cakes on the bottom in future revisions? Out of scope here; flagged for the design team to confirm during the next content review.
